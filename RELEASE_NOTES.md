## v1.0.3

本版本修复部分问题电脑上“备用读取”仍无法读取当前单位的问题。根因是这些电脑的 Warcraft III 会把真实单位对象和 native 表分配到 4GB 以下的虚拟地址，而旧版把所有低于 `0x100000000` 的地址一律判定为无效指针。

### 变更

- 备用读取不再使用固定的 4GB 指针下限，改为根据当前 Warcraft III 进程实际已提交、可读的内存区域验证地址。
- 低地址只有确实落在当前进程可读区内才会被接受；JASS handle、普通整数和未映射地址不会被误当成单位指针。
- 修复低地址 native 字符串无法解码的问题，避免问题机缺少技能字段等 native handler。
- 普通读取路径保持不变；低地址兼容仅用于“备用读取”及其后续字段读取和写入。
- 备用日志新增 resolver 地址、解析后单位地址的可读性及所属内存区域，便于继续诊断其他布局。

### 证据与验证

- 新问题机三次日志均成功取得实时 JASS 选择句柄 `0x100656`，并稳定解析出 `0x82b70cc8`；旧版仅因该真实地址低于 4GB 而拒绝继续映射。
- 低地址回归测试确认 `0x82b70cc8` 位于模拟可读区时会被接受，而 `0x100656` 仍不会被识别为指针。
- 低地址 native 字符串回归通过。
- 本机高地址布局下，普通读取与备用读取均读取到同一单位、相同 HP/MP 和完整 `77` 个字段；备用 native 恢复为 `86/86`。
- 备用身份二次读取返回相同 handle、owner、unit 和字段数量。

### 兼容性

- 已按 Warcraft III: Reforged `2.0.4.23745` 验证。
- 遇到 `WinError 299`、字段缺失、错误单位、锁定旧单位或低地址布局时，使用普通读取旁边的“备用读取”。

## v1.0.2

本版本新增技能字段临时编辑，并为部分电脑出现的 `WinError 299`、字段缺失、读取错误单位和锁定第一次读取单位的问题提供独立的“备用读取”兼容路径。

### 遇到读取异常时怎么操作

1. 先在游戏中单击并只选中需要修改的目标单位。
2. 如果普通 `读取当前选中单位` 出现 `WinError 299`、字段不全、读取失败、读到错误单位，或一直停留在之前的单位，请点击旁边的 `备用读取`。
3. 备用读取成功后，继续在同一字段表中修改属性、智力、技能和物品栏；后续写入会沿用备用兼容路径。
4. `备用读取` 不仅适用于 Win10。Win11 或其他存在不同内存分页、堆布局的电脑遇到同类问题时也应使用它。
5. 若备用读取仍失败，请提供修改器同目录 `log\win10-read-latest.log`；程序目录不可写时，日志位于系统临时目录 `War3ReforgedTrainer\log`。

### 变更

- 新增独立“备用读取”按钮，并放到普通读取按钮旁边；原普通读取路径保持不变。
- 备用读取使用更容错的分区读取和 native 恢复路径，能够跳过不可读页和部分复制页，适配不同电脑上的进程内存布局。
- 备用读取会通过 JASS 实时选择结果和 `IsUnitSelected` 交叉校验当前目标，不再允许历史 handle 或上一次读取的单位单独兜底，修复“读过步兵后后续一直锁在步兵”的问题。
- 备用读取下的智力读取和写入改为直接调用 `GetHeroInt` / `SetHeroInt`，并在写入后读回验证。
- 新增独立“技能字段”页：输入技能 rawcode 和字段等级后，可读取当前技能实例中已开放的整数、实数、布尔和字符串字段，并对选中字段进行临时写入和读回校验。
- 保留完整字段读取、普通读取、大象功能、快捷键、技能与物品栏等既有路径。

### 已验证

- 问题电脑已确认备用读取能够读取真实当前英雄，不再锁定第一次读取的步兵，并能返回完整字段。
- 备用 native 配置恢复结果为 `27/27`，实时选择走 `jass_no_sync_verified` 路径。
- 备用智力写入完成 `41 -> 42 -> 41` 的写入、读回和恢复验证。
- `python -m py_compile .\war3_reforged_trainer.py .\war3_ability_fields.py`、Ruff 和 `git diff --check` 通过。
- 最终单文件 EXE 已确认包含技能字段模块、native helper 和程序图标。

### 兼容性

- 已按 Warcraft III: Reforged `2.0.4.23745` 的当前内存结构验证。
- 普通读取正常的电脑继续使用普通读取；只有出现上述异常时才需要使用备用读取。

## v1.0.0

本版本完成“大象功能”与全局快捷键体系，并将经过当前 Warcraft III 重制版实机验证的功能作为首个正式版本发布。

### 变更

- 新增独立“大象功能”页，覆盖地图、游戏、英雄、单位、物品、技能、科技、资源、增益/减益与全屏效果等操作。
- 新增“快捷键功能”页，共 45 个全局快捷键；每项可独立启用，危险操作默认关闭，系统占用键会明确显示冲突。
- 修复英雄等级只能降低、无法提高的问题；设置后会读回验证实际等级。
- 修复新增技能后技能等级写入无效的问题。
- 直接技能效果不再创建可见假单位；临时技能由游戏引擎创建，并在执行后验证删除，不会留下额外单位或技能实例。
- 为持续技能增加停止结果校验、有限重试、作用范围精确恢复和异常清理。
- helper 协议升级至 v15，增加完整跨进程事务、对象身份复核、内部调用异常保护和全图枚举截止时间。
- GUI 关闭时会等待正在执行的后台操作完成清理，避免中途退出留下临时状态。
- 显示当前选中单位 ID，并保留完整字段读取、当前人口写入及资源锁定等既有功能。
- 更新程序图标、窗口标题和 Windows 文件版本信息为 `1.0.0`。

### 已验证

- Warcraft III: Reforged `2.0.4.23745` 实机发现并验证 `71/71` 个所需 native handler。
- 45 个快捷键与 45 个回调一一对应，无“待适配”占位。
- `AOsh` 添加、等级 `1 -> 2`、删除均成功，读回结果一致。
- `Ainf`、`ANht`、`AEtq` 直接效果与临时实例清理成功；`AEtq` 连续执行三次均正确停止并恢复。

- 最终单文件 EXE 已验证内嵌 helper、程序图标、快捷键页、后台安全关闭及游戏进程响应状态。

## v0.2.8

本版本修复部分游戏局次或单位内存分配下，选中英雄只能读取基础与攻击字段，英雄属性、技能和物品栏字段缺失的问题。

### 变更

- 组件发现不再假定 `move/attack/hero/inventory` wrapper 位于单位 owner 附近。
- 首次读取时一次扫描并建立全进程组件 owner 索引，通过组件标签、owner 指针、wrapper/data vtable 和 data 指针共同校验。
- 同一游戏进程内缓存所有单位的组件索引；切换其他单位后直接按 owner 读取，地址失效或出现新 owner 时自动重建。
- 保留邻域扫描作为快速路径，并修复 wrapper 跨相邻内存区域边界时被漏掉的问题。
- 无组件单位会记录为已检查，不会因为索引中没有组件而重复执行全局扫描。

### 已验证

- `python -m py_compile .\war3_reforged_trainer.py`
- 在 Warcraft III `2.0.4.23745` 当前局的 `103` 个单位上，将新索引与独立逐标签全内存扫描逐一对照，组件集合不一致数量为 `0`。
- 当前局组件计数完全一致：`attack=89`、`move=89`、`hero=1`、`inventory=1`。
- 当前剑圣读取到 `73` 个字段，其中英雄 `8`、技能 `20`、物品栏 `12`、能力实例 `4`；第一物品槽为 `stwp`、数量为 `1`。
- 无组件、普通 `attack+move`、英雄 `attack+hero+inventory+move` 三种当前局实际结构均生成与其组件相符的字段集合。
- 首次全局索引约 `9.6-11.3` 秒；同一进程后续完整字段读取约 `124-130ms`，无组件单位缓存读取约 `3ms`。
- 修正版打包 exe 已在 GUI 中点击“刷新字段表”验证，字段表实际显示 `物品槽1=stwp`、`物品槽1数量=1`。

### 限制

- 普适性验证范围是 Warcraft III `2.0.4.23745` 当前内存结构；暴雪更新对象结构后仍可能需要重新适配。
- 首次读取需要建立全局组件索引，耗时会明显高于同一游戏进程内的后续读取。
